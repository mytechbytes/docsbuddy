)

3.3 — Enforcing it with Postgres RLS

One helper function + one policy per table replaces hand-rolled middleware. Supabase runs every query under auth.uid(), so the database itself becomes the security boundary.

create or replace function is_family_member(fid uuid, min_role family_role default 'viewer')
returns boolean language sql stable as $$
  select exists (
    select 1 from family_members fm
    where fm.family_id = fid
      and fm.user_id = auth.uid()
      and case min_role
            when 'viewer' then true
            when 'member' then fm.role in ('owner','admin','member')
            when 'admin'  then fm.role in ('owner','admin')
            when 'owner'  then fm.role = 'owner'
          end
  );
$$;

alter table assets enable row level security;

create policy "read assets in my family" on assets
  for select using (is_family_member(family_id));

create policy "write assets as member+" on assets
  for all using      (is_family_member(family_id, 'member'))
            with check (is_family_member(family_id, 'member'));

3.4 — Invitation flow

  1. Owner/admin generates an invite → row in family_invites with a short code.
  2. App produces a deep link (docsbuddy://join?c=AB12CD) + a fallback web URL.
  3. Invitee opens the link → if signed in, app calls rpc.accept_invite(code); if not, sign-in first.
  4. RPC validates expiry, inserts into family_members, marks invite accepted, atomically.

Document storage architecture

PDFs, images, and receipts attach to assets. Files live in object storage; only metadata sits in Postgres. Access is gated by the same family RBAC, with short-lived signed URLs for downloads.

4.1 — Storage layout

📁 docsbuddy-files/ # single private bucket 📁 {family_id}/ # top-level folder = tenant boundary 📁 assets/ 📁 {asset_id}/ 📁 documents/ 📄 {doc_id}_invoice.pdf 📄 {doc_id}_warranty.pdf 📁 thumbnails/ 🖼 {doc_id}_w400.jpg # generated by edge fn 📁 locations/ 🖼 {location_id}_cover.jpg

The leading {family_id} segment lets a single Storage RLS policy enforce isolation:

create policy "family members read their files" on storage.objects
  for select using (
    bucket_id = 'docsbuddy-files'
    and is_family_member((storage.foldername(name))[1]::uuid)

Section 05

Intelligent notification pipeline

A single daily cron computes who should hear about which reminder, dispatches a push to every relevant device, and is idempotent — running twice on the same day never produces duplicate alerts.

Reminders live on the device. The phone schedules its own local notifications and never depends on the network to fire them. The server is a sync target — it stores the canonical copy and fans changes out to other family members' devices, WhatsApp-style.

5.1 — Threshold model

Each asset_dates row carries a per-reminder array of offsets in days. Defaults are configurable in user settings and propagate to new reminders.

−30 days
First nudge
−7 days
Reminder
−1 day
Final
Due
Date

Users can override globally (Settings → Notifications) or per-reminder (e.g. "remind me 60/14/3 days before insurance").

5.2 — The daily worker

A Supabase Scheduled Edge Function (or a Cloud Function / EventBridge rule on other clouds) runs every hour on the hour. At each tick it processes users whose local time is 08:00.

// pseudocode for the scheduled worker (TypeScript / Deno on Supabase)
export default async function tick() {
  const nowUtc = new Date();

  // 1. Find users whose local clock is 08:00 right now.
  const users = await db.users.where(localHour(timezone, nowUtc) === 8);

  // 2. Pull reminders that hit a threshold today, for any family those users belong to.
  const rows = await db.raw(`
    select ad.*, a.name as asset_name, a.family_id, fm.user_id, ud.fcm_token, ud.platform,
           (ad.due_date - current_date) as days_left
    from asset_dates ad
    join assets a            on a.id = ad.asset_id
    join family_members fm   on fm.family_id = a.family_id
    join user_devices ud     on ud.user_id = fm.user_id
    where ad.completed_at is null
      and (ad.due_date - current_date) = any(ad.notify_offsets)
      and (ad.last_notified_offset is null
           or ad.last_notified_offset > (ad.due_date - current_date))
      and fm.user_id = any($1)
  `, [users.map(u => u.id)]);

  // 3. Group by reminder; one push per device with a payload that the app deep-links.
  for (const r of rows) {
    await fcm.send({
      token: r.fcm_token,
      notification: {
        title: `${r.asset_name} — ${r.label} in ${r.days_left} days`,
        body:  `Due ${formatDate(r.due_date)}`
      },
      data: { asset_id: r.asset_id, asset_date_id: r.id, deep_link: `docsbuddy://asset/${r.asset_id}` },
      apns: { headers: { 'apns-priority': '10' } }
    });
  }

  // 4. Mark each reminder so we don't re-fire the same threshold.
  await db.asset_dates.markFired(rows);
}

5.3 — Why last_notified_offset?

It makes the worker idempotent. If the cron retries, or the user's timezone hops daylight savings, we never resend the same nudge. The clause last_notified_offset > days_left only matches strictly earlier thresholds.

5.4 — Local notifications as a fallback

On every login/sync, schedule a small ring buffer (next ~30 days of reminders) as local notifications using flutter_local_notifications. This guarantees that even if the device is offline when the FCM push fires, the user still gets reminded — and you get a free graceful degradation if FCM ever has an outage.

5.2 — On-device scheduler

When a reminder is created or edited, the app computes every threshold date and schedules an OS-level local notification for each one. No server round-trip required — the alert fires even in airplane mode.

// services/reminder_scheduler.dart
Future<void> rescheduleFor(Reminder r) async {
  await _plugin.cancelAll(tag: 'r:${r.id}');
  for (final offset in r.notifyOffsets) {
    final when = r.dueDate.subtract(Duration(days: offset));
    if (when.isBefore(DateTime.now())) continue;
    await _plugin.zonedSchedule(
      id:   stableInt(r.id, offset),
      tzTime: tz.TZDateTime.from(when, tz.local).withHour(8),
      title: '${r.assetName} — ${r.label} in $offset days',
      body:  'Due ${formatDate(r.dueDate)}',
      payload: 'docsbuddy://asset/${r.assetId}',
      tag:   'r:${r.id}',
      androidScheduleMode: AndroidScheduleMode.exactAllowWhileIdle,
    );
  }
}

iOS & Android allow ~64 scheduled local notifications per app, plenty for hundreds of reminders since most are months out — only the next ~30 days are actually queued at a time. A background task (workmanager) re-fills the queue weekly so the schedule stays current.

5.4 — Conflict resolution

Two devices can edit the same reminder offline. On sync, the server compares version + updated_at and resolves last-write-wins per field (not per row) so changes from both edits survive when they touch different fields. Tombstones (deleted_at) win against any concurrent edit — deletes are sticky for 30 days, then garbage-collected.

// server-side merge — runs inside a transaction
function merge(server, incoming) {
  if (server.deleted_at || incoming.deleted_at)
    return { ...server, deleted_at: server.deleted_at ?? incoming.deleted_at };

  const out = { ...server };
  for (const field of EDITABLE_FIELDS) {
    const sStamp = server.field_versions[field]   ?? 0;
    const iStamp = incoming.field_versions[field] ?? 0;
    if (iStamp > sStamp) out[field] = incoming[field];
  }
  out.version    = Math.max(server.version, incoming.version) + 1;
  out.updated_at = new Date();
  return out;
}

5.5 — Files: chunked, deduped, lazy

Invoices, receipts and screenshots follow the WhatsApp media model — the file's bytes and its database row sync independently. The row arrives first (small), the bytes follow when bandwidth allows.

5.6 — When the server does push

The only push notifications the server sends are silent data pushes to wake the app and trigger a sync. The phone, not the server, decides what alerts to actually display. This means:

  • If your laptop edits a reminder, all family devices get a silent FCM ping → background sync → local OS notification ("Anand changed insurance date to 17 Jun").
  • If you're offline when the threshold passes, the local alarm still fires.
  • If the server is down, the app keeps working. Sync just queues up.

6.4 — Offline strategy

The repository layer is the seam. Every repository reads from Drift first, returns a stream, and reconciles with Supabase in the background.

  1. Read — UI subscribes to a Drift stream. Latest local data renders immediately.
  2. Sync — On startup, on pull-to-refresh, and on Supabase realtime events, repository fetches deltas (rows where updated_at > last_sync) and upserts into Drift.
  3. Write — Mutations write to a local outbox table first, then attempt remote. On success they're cleared; on failure they retry with exponential backoff.
  4. Files — Documents are downloaded on-demand to the cache directory; recently-viewed docs survive offline. Uploads queued in the outbox carry a local file path until they succeed.

class UpcomingReminders extends _$UpcomingReminders {
  @override
  Stream<List<ReminderView>> build() {
    final repo = ref.watch(reminderRepositoryProvider);
    return repo.watchUpcoming(within: const Duration(days: 90));
  }

  Future<void> markDone(String id) async {
    await ref.read(reminderRepositoryProvider).markCompleted(id);
  }
}
